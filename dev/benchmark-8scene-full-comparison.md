## 八、整体结论

### 8.1 核心数字

```
                   基准（国内）   当前（Google）
                   ───────────   ─────────────
总搜索              57 次         28 次（−51%）
搜索耗时            1,079.7s      488.2s（−55%）
fetch 耗时          22.5s         84.2s（+274%）
🎯 总耗时           1,102s        572s（−48%）
单次搜索            18.9s         19.5s（±3%）
单次 fetch          3.2s          8.4s（+163%）
迭代轮次            25 轮         21 轮（−16%）
失败调用            0 次          5 次
噪声场景            3/8            0/8
准确匹配            6/8            6/8
🐛 词汇桥断裂        1/1（台风）   0.5/1（半修复）
```

### 8.2 谁更好？

| 维度 | 胜出方 |
|------|:---:|
| 搜索速度（单次） | **平手**（19.5s vs 18.9s） |
| 总耗时 | **Google**（−48%，搜得少不是搜得快） |
| fetch 速度 | **国内**（+163%，国内站延迟优势明显） |
| 噪声率 | 🔴 **Google 碾压**（0% vs ~20%） |
| 稳定性 | **国内**（0 失败 vs 5 失败） |
| 中文内容覆盖深度 | **国内**（baidu/sogou 长尾中文源） |
| 最新信息 | **Google**（版本号更新、旅游噪声消除） |
| SEO 污染免疫 | 🔴 **Google 碾压**（bing_cn 极度脆弱） |
| 词汇桥 | **Google 略优**（但都未完全解决） |
| 实时数据 | **平手**（都依赖 fetch 突破） |

### 8.3 关键发现

1. **单次搜索速度无差异**：Google 不更快。Playwright 浏览器渲染耗时抵消了 Google 后端的任何速度优势。总耗时减半纯粹因为搜得少。

2. **噪声率是最显著的差异维度**：Google 零噪声（0% vs 基准 ~20%），尤其是场景 3（上海 GPU）bing_cn 的 80% 旅游污染在 Google 上完全消失。这一项就节省了 16% 的迭代轮次。

3. **搜索引擎索引延迟是共同瓶颈**：场景 8（台风）两个环境都只能靠 fetch nmc.cn 获取实时数据——搜索引擎索引周期（小时→天）与气象数据更新频率（3-6 小时）根本不在同一量级。

4. **词汇桥问题在 Google 上部分缓解但未解决**：已知英文台风名后 Google 能找到第三方报道，但 sogou/baidu/bing_cn 完全找不到。根本原因是中文搜索引擎做的是词法匹配而非语义理解。

5. **Docker+VPN 的 fetch 代价很高**：翻墙访问国内站（typhoon.nmc.cn、cnblogs.com、IT之家）延迟 2-7 倍。对以中文内容为主的查询场景（场景 3/5/6/8），这个代价不可忽视。
